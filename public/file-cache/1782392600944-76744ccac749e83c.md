# 用 AI 做投资研究 SOP · Skill 草稿

## 这是什么

面向 PE/VC/FA 投研人员的**用 AI（Claude Code + Cursor + Obsidian + 联网工具）搭建投资研究工作流的实操 SOP**。2026 年最大的投研生产力跃迁不是单个 AI 工具，而是用 Claude Code 把"数据抓取→知识库沉淀→分析建模→报告撰写"全流程串起来——让 AI 从"聊天助手"升级为"执行者"。但多数投研人仍把 AI 当聊天机器人用（一问一答），不知道 Claude Code 与网页版的本质区别、不会搭知识库、不会配联网工具、不会写规则文件。本 Skill 提供一套可直接照搬的配置方案：Claude Code + Cursor（载体）+ Obsidian（知识库）+ Playwright/Firecrawl（联网），可作为任何投研团队的 AI 增强基础设施。适合投研团队、独立分析师、FA、PE/VC 分析师。

> 与本系列其他 Skill 的关系：本 Skill 是"基础设施"——前面 5 份（政策/论文/专利/专家/招投标）检索方法论都可以**作为 AI Agent 的任务指令**嵌入本工作流，让 AI 自动执行这些检索任务。

## 方法论框架（6 维度）

### 维度 1：核心认知——Claude Code ≠ 聊天机器人

> 多数投研人望而却步的原因：Claude Code 是黑漆漆的命令行界面，看起来是工程师的工具。这是一个致命误解。

#### Claude（网页版）vs Claude Code 的本质区别

| 维度 | Claude 网页版 | Claude Code |
|---|---|---|
| **本质** | 聊天机器人，一问一答 | 拥有实操能力的 Agent |
| **记忆** | 单次对话，关掉就忘 | 可读写本地文件 = 长期记忆 |
| **工具** | 不能联网/不能开浏览器 | 可开浏览器、抓数据、读写文件 |
| **流程** | 每步需手动衔接（复制粘贴） | 可独立跑完整流程 |
| **载体** | 浏览器窗口 | Cursor / 终端 |

**核心隐喻**：Claude 网页版是"供你交流思考的大脑"，Claude Code 是"能帮你落地整套工作流的执行者"。名字里的"Code"是最大的误导——它不只是编程工具。

#### 投研场景对比

- **网页版做投研**：选股 → 粘贴财报 → 搭模型 → 润色报告 → 每步手动衔接。
- **Claude Code 做投研**：下达目标"筛选新能源车产业链、按内部分析框架精简、选 2 家深度分析、抓财报建模型、压力测试、估内在价值、写研报"→ AI 自动跑完整流程，仅在需要人工介入时询问。

### 维度 2：工具栈搭建（4 件套）

#### 工具 1：Claude Code（核心引擎）

- **安装**：Anthropic 官网 code.claude.com/docs/en/quickstart，Mac/Windows 均可。
- **启动**：终端输入 `claude` → 进入交互界面 → 自然语言提问。
- **订阅**：需 Claude Pro/Max 订阅。
- **替代方案**：MiniMax M2 可接入 Claude Code 作为底层引擎，成本约原版 7%（无订阅时可选）。

#### 工具 2：Cursor（核心载体）

- **作用**：不是替代命令行，而是**工作体系的核心载体**。
- **价值**：在一个窗口同时看：知识库文件夹 + Claude Code 对话 + 文件创建/编辑过程。
- **使用**：将 Obsidian 笔记库文件夹在 Cursor 中打开 → 侧边栏可见所有文件 → Claude Code 在侧面板运行 → 实时看到文件创建/编辑/索引。

#### 工具 3：Obsidian（知识库）

- **作用**：解决 Claude Code 无记忆的问题——把电脑文件夹打造成可不断沉淀的个人研究资料库。
- **特点**：免费、纯文本（Markdown）、笔记互联、全局检索、本地存储、不绑定平台。
- **核心用法**：让 Claude Code 读写同一个笔记文件夹 → 知识库成为它的长期记忆。
- **价值**：每次问答结果归档 → 知识库持续迭代优化 → 3 月记录的半导体笔记 11 月仍可调取。

#### 工具 4：Playwright + Firecrawl（联网能力）

> 前三个工具只在本地知识库内工作，投研需联网抓实时数据——这两个免费开源工具实现联网。

- **Playwright**（微软开发）：Claude Code 可模拟人工操作浏览器——打开网页、点击按钮、填表单、登录账号、读取页面。金融场景中选股平台设筛选条件、财报藏在下拉菜单、监管平台需同意免责声明才能下载——普通爬虫无法应对，Playwright 完美模拟人工。
- **Firecrawl**：网页内容抓取转 Markdown，适合批量文档化。

### 维度 3：知识库目录结构（搭骨架）

> 在 Obsidian 笔记库根目录建 3 个文件夹 + 1 个总目录文件。

```
笔记库根目录/
├── raw/                    # 原始素材接收区（剪藏/导入的原文）
├── wiki/                   # 核心知识库（Claude Code 撰写维护）
│   └── _master-index.md    # 总目录（所有主题的入口）
├── output/                 # 查询结果与正式报告
└── CLAUDE.md               # 规则文件（最重要！）
```

**三目录分工**：
- `raw/`：素材收纳箱——文章、文稿、截图、临时笔记都放这，无需整理。
- `wiki/`：核心知识库——由 Claude Code 撰写结构化摘要、分类归档、维护全局目录。
- `output/`：最终产出——研究报告、分析结论，与原始资料、知识库区分。

**主题组织**：每个独立主题在 `wiki/` 下单独建子文件夹 + `_index.md`（主题目录）。例：`wiki/半导体/`、`wiki/新能源车/`。

### 维度 4：CLAUDE.md 规则文件（最关键）

> 每次在笔记库启动 Claude Code，它会先读这个文件——规则只写一次，永久生效。没有它 Claude Code 只是普通助手，有它就是规范化的资料管理员。

#### CLAUDE.md 完整模板（可直接复制）

```markdown
# 知识库管理规则

1. 本知识库由大语言模型负责维护，你的身份是资料管理员。
2. wiki/ 目录由你全权负责撰写与维护，我极少直接编辑该目录下的文件。
3. raw/ 为原始素材目录。我将文件存入此处后，你需要在"整理归档"环节将素材处理并归入知识库。
4. wiki/_master-index.md 是总入口，需罗列所有主题文件夹并附单行简介，保证内容实时更新。
5. 每个独立主题在 wiki/ 下单独建子文件夹，每个主题文件夹内新建 _index.md，罗列该主题下所有文章并附简要说明。
6. 统一使用文档内部链接关联不同主题的相关内容。
7. 整理原始素材流程：
   - 读取 raw/ 目录内的文件
   - 判定归属主题，无对应主题则新建文件夹
   - 撰写知识库文章，提炼核心要点并关联相关文档
   - 更新对应主题的 _index.md
   - 更新全局总目录 wiki/_master-index.md
   - 若一份素材涉及多个主题，需在对应主题下分别建文并互相链接
8. 文章力求简洁，优先使用项目符号，减少大段文字叙述。
9. 每篇知识库文章必须增设"核心要点"板块。
10. output/ 专门存放查询结果与生成报告。
11. 回答问题前，先查阅全局总目录，再深入对应主题目录，最后调取具体文章内容作答。
12. 我下达"整理归档"指令时，处理 raw/ 内所有未归档的素材。
13. 我下达"核查纠错"指令时，检查知识库内容矛盾、链接失效、内容缺失等问题，并给出优化建议。
```

### 维度 5：日常工作四步流程（怎么用）

> 整套体系搭建完成后，日常使用只需 4 个核心操作。

#### 步骤 1：剪藏（素材收集）

- 浏览器装 Obsidian Web Clipper 插件（obsidian.md/clipper）。
- 模板设置：存储位置 = `raw/`，命名格式 = `{{date|date:"YYYY-MM-DD"}}-{{title|safe_name}}`。
- 看到有价值内容一键剪藏至 raw/ → 无需整理、无需写摘要 → 仅做收纳。
- 可选：装 Local Images Plus 插件 → 自动下载图片至本地，网页失效仍保留。

#### 步骤 2：整理归档（定期）

- 素材堆积较多时（每周或更频）在 Claude Code 输入：

```
将 raw/ 目录内所有素材整理归档至知识库。处理要求：
1. 读取每份文件，判定核心主题；
2. 在 wiki/ 目录中匹配或新建对应主题文件夹；
3. 撰写文章摘要、提炼核心要点，并添加文档内链接关联相关内容；
4. 更新对应主题目录文件；
5. 更新全局总目录。
所有相关主题之间做好内容关联。
```

- Claude Code 自动完成全文解读、分类、写摘要、更新目录——一杯咖啡时间清空素材箱、扩充知识库。

#### 步骤 3：查询调取（高频）

3 类用法：
- **简单查询**："上个月我归档的半导体行业报告有哪些核心结论？"
- **交叉分析**："中国汽车出口放缓，和大宗商品需求之间有哪些关联？"
- **深度总结**："结合知识库所有内容，分析印度特种化学品行业的多空逻辑，将结论新建为知识库文章，并标注参考来源。"

第三类让知识持续沉淀——每次问答生成新笔记、关联原有资料，知识库越用越完善。

#### 步骤 4：核查优化（月度）

每月一次，输入：

```
核查整个知识库，梳理内容矛盾、链接失效、交叉引用缺失、内容空白等问题，暂不修改内容，仅出具核查报告。
```

Claude Code 全面巡检、指出问题，由你决定如何优化。

### 维度 6：投研专属增强（把检索 Skill 嵌入）

> 把本系列其他 5 份检索方法论（政策/论文/专利/专家/招投标）做成"任务模板"嵌入知识库——AI 可自动执行这些检索任务。

#### 嵌入方式

在 `wiki/_templates/` 下建任务模板：

```markdown
# 任务模板：政策追踪

## 触发词
"追踪 [行业] 政策"

## 执行步骤
1. 按《政策检索方法论》维度 1 确定核心监测源（国务院+部委+地方）。
2. 用 Playwright 打开 gov.cn / 对应部委官网。
3. 按维度 2 高级搜索运算符检索关键词 + 时间范围。
4. 按维度 4 八维度框架解读关键政策。
5. 在 wiki/[行业]/ 下生成政策追踪文章，更新 _index.md。
```

#### 投研典型工作流（Claude Code 可执行）

- **个股筛选**："从全市场筛选新能源车产业链标的，按 [内部框架] 精简到 10 家" → Playwright 抓数据 + Claude Code 套框架。
- **深度分析**："对 [公司 A] 做深度分析，抓最近 3 年财报，建 DCF 模型，压力测试"。
- **行业研究**："做半导体设备行业研究，按《半导体投资框架》6 维度展开，调用 wiki/ 内已有资料"。
- **政策追踪**："追踪本周氢能政策动态，按《政策检索方法论》执行"。
- **论文监测**："查 arXiv 上本周 [领域] 新论文，识别可能影响 [被投公司] 的突破"。
- **报告撰写**："基于以上分析，撰写 IC 备忘录，存入 output/"。

## 行业核心指标

| 指标 | 健康阈值 | 说明 |
|---|---|---|
| **知识库主题数** | 持续增长 | 覆盖广度 |
| **单主题文章数** | > 10 篇 | 深度积累 |
| **每周归档素材** | > 20 篇 | 持续输入 |
| **交叉引用密度** | 每篇平均 > 3 个内部链接 | 知识关联度 |
| **检索响应** | < 1 分钟调取历史资料 | 沉淀价值兑现 |
| **月度核查修复** | 矛盾/失效 < 5% | 知识库质量 |
| **联网工具覆盖** | Playwright + Firecrawl 均可用 | 实时数据获取 |

## 红旗识别
1. **把 Claude Code 当聊天机器人用**——只一问一答，不搭知识库 = 浪费 90% 价值。
2. **不建目录结构**——所有文件堆一个文件夹，无法管理。
3. **不写 CLAUDE.md 规则文件**——AI 没有规范，归档质量参差不齐。
4. **不定期整理归档**——raw/ 堆积如山，知识库不更新。
5. **不做月度核查**——矛盾、失效链接累积，知识库腐化。
6. **不装联网工具**——只有本地资料，无法获取实时数据。
7. **不设交叉引用**——笔记孤立，无法做关联分析。
8. **只用一家 AI 工具**——不同模型有不同优势（Claude 深度思考 + ChatGPT 快速 + Cursor 编码），应组合。
9. **把 AI 结论当权威**——AI 可能编造，关键结论必须交叉验证。
10. **不沉淀工作流模板**——重复任务每次重新描述，效率低。
11. **知识库无主题分层**——所有主题混在一起，无法管理。
12. **CLAUDE.md 规则过于复杂**——规则 > 20 条 AI 执行偏差大，应精简。

## 占位符
- `[行业/主题]`：被研究的行业或主题（半导体/新能源/生物医药等）
- `[内部框架]`：团队的分析框架（可放在 wiki/_frameworks/）
- `[任务模板]`：政策追踪/论文监测/专利分析/专家访谈/招投标验证
- `[查询类型]`：简单查询/交叉分析/深度总结
- `[输出格式]`：IC 备忘录/行业报告/政策简报/个股分析
