# Contributing to N.E.I. / nei-pevc.com

欢迎参与 N.E.I.。这个项目面向 PE / VC / FA / 产业投资等一级市场从业者，目标是把高频投资工作流沉淀成可搜索、可收藏、可通过 MCP 调用的 AI Skill 与 Workflow。

如果你是第一次用 GitHub，不用紧张。你不需要一开始就会写代码，也可以贡献内容、场景和使用反馈。

## 当前维护模式

`nei-pevc.com` 目前采用“主维护者负责制 + 社群贡献”的轻量模式：

- **主维护者**：lensnowovo，负责产品方向、部署、安全、MCP、代码合并和线上发布。
- **社群贡献者**：可以通过网站投稿、GitHub Issue 或 Pull Request 贡献内容和代码。
- **GitHub Maintainer Agent**：自动为 Issue 打标签、回复下一步说明，并为 PR 运行基础质量检查。
- **最终发布权**：上线到 `nei-pevc.com` 的代码、内容和 MCP 能力由主维护者审核后发布。

这个模式的目标是保持速度，同时避免多人直接改线上系统带来的安全和质量风险。

## 你可以怎么贡献

### 1. 贡献内容

适合 PEVC / FA / 产业投资从业者。

可以贡献：

- BP 初筛 Prompt
- 行业研究 Workflow
- 商业尽调问题清单
- IC Memo 模板
- 投后月报模板
- 募资 / LP 沟通材料
- 政府引导基金、公文、正式函件模板
- MCP 使用案例

推荐路径：

1. 优先在网站投稿；
2. 如果还不确定格式，在 GitHub 新建 Content Contribution Issue；
3. 维护者会判断是否进入公开库、是否 MCP Ready、是否需要补输入 / 输出示例。

### 2. 反馈 bug

请新建 Bug Report Issue，并尽量写清楚：

- 发生在哪个页面；
- 你点了什么；
- 期待看到什么；
- 实际发生了什么；
- 是否登录；
- 浏览器和设备；
- 如果方便，附截图。

### 3. 提产品建议

适合提出：

- 新页面；
- 新 Skill 分类；
- MCP 新工具；
- 管理后台能力；
- 安全与隐私提示；
- 内容质量规则。

请尽量用真实工作场景描述，例如：

> 我希望在写 IC Memo 前，系统能根据收藏 Skill 自动推荐一个“行业研究 → 商业尽调 → 风险质询 → Memo 结构”的 Bundle。

### 4. 贡献代码

适合熟悉 Next.js / TypeScript / Prisma / MCP 的贡献者。

建议流程：

1. 先开 Issue 说明要改什么；
2. 等维护者确认方向；
3. fork 或新建分支；
4. 提交 Pull Request；
5. 等待 review。

不要在未沟通的情况下提交超大 PR。N.E.I. 当前处于 Public Beta 前阶段，小而清晰的改动更容易合并。

## 代码质量要求

提交 PR 前请至少运行：

```bash
npm run lint
npx tsc --noEmit --pretty false
npm run build
```

如果改动涉及公开 Skill 详情页，请运行：

```bash
npm run smoke:public-posts
```

## 内容质量要求

一条好的 Skill / Workflow 至少应该说明：

- 适用角色；
- 使用场景；
- 输入材料；
- 操作步骤；
- 输出结果；
- 示例输入；
- 示例输出；
- 安全边界；
- 不适用场景。

首页精选和 MCP 准入会优先考虑结构完整、场景明确、质量分较高的内容。

## 安全规则

请不要提交：

- API Key；
- 数据库连接串；
- MCP Token；
- 用户 BP、财务模型、投委会材料、LP 名单；
- 任何未经脱敏的真实项目材料；
- 会读取本地敏感文件的 MCP / Agent 代码；
- 未经审核的外部脚本或可疑依赖。

如果你发现安全问题，请不要公开贴出可利用细节。可以先联系维护者或创建一个只描述影响范围的 Issue。

## 分支和发布

- `main`：正式主分支，连接线上部署。
- 普通贡献请走 Issue / Pull Request。
- 线上发布由主维护者确认。
- Vercel、域名、数据库、生产环境密钥由主维护者管理。

## 给非工程贡献者的一句话

你不需要会写代码。只要你能清楚描述一个真实的 PEVC 工作任务、输入材料和理想输出，它就可能被整理成 N.E.I. 的一个 Skill 或 Workflow。
